# Source Code Audit — 2025-05-21

## 🔴 High Severity

### 1. TOCTOU race condition in duplicate check

- **File:** `src/features/recurring/application/use-cases/execute-recurring-rule.use-case.ts:43-55`
- **Issue:** Two concurrent executions can both pass `hasTransactionForRuleDate` before either inserts, creating duplicate transactions for the same rule+date.
- **Fix:** Database-level unique constraint on `(recurring_id, date)` where `deleted_at IS NULL`, or use an advisory lock / serialized execution.

### 2. Partial rollback creates unrecoverable duplicate on retry

- **File:** `src/features/recurring/application/use-cases/execute-recurring-rule.use-case.ts:76-92`
- **Issue:** If `setAccountBalance` succeeds but `updateExecutionState` fails, the catch block soft-deletes the transaction and reverts the balance, but does **not** advance `next_due_date`. On next execution the duplicate check finds the soft-deleted transaction → returns `false` → creates a _new_ transaction. Repeat on each failure.
- **Fix:** In the rollback, also revert the execution state change (or ensure idempotency via the unique constraint from #1).

### 3. Delete/soft-deactivate server actions lack try/catch

- **Files:**
  - `src/features/transactions/server/actions.ts:68-83`
  - `src/features/recurring/server/actions.ts:73-101`
  - `src/features/accounts/server/actions.ts:72-85`
  - `src/features/categories/server/actions.ts:72-85`
- **Issue:** Create/update actions have proper error handling; delete actions don't. DB errors → unhandled 500. Also return `Promise<void>` instead of `Promise<ActionState>`, making them incompatible with `useActionState`/`useServerActionForm`.
- **Fix:** Add try/catch to all delete actions, return consistent `ActionState` type.

## 🟡 Medium Severity

### 4. Floating-point precision in Money arithmetic

- **File:** `src/shared/domain/value-objects/money.ts:40-46`
- **Issue:** `Money.add()`/`.subtract()` use raw JS `number`. `0.1 + 0.2 = 0.30000000000000004`, accumulates across transactions.
- **Fix:** Use integer cent/paise representation or a decimal library like `decimal.js`.

### 5. `formatTransactionGroupDate` uses UTC for today comparison

- **File:** `src/features/transactions/domain/format.ts:38-48`
- **Issue:** `new Date().toISOString().slice(0, 10)` produces UTC date. In UTC+5:30 at 3 AM local, `todayKey` = yesterday. Today's transactions won't match, so "Today" label never appears in positive timezones.
- **Fix:** Use `new Intl.DateTimeFormat().format()` or manual local date extraction instead of `.toISOString()`.

### 6. Locale/currency duplicated in format file

- **File:** `src/features/transactions/domain/format.ts:1-12`
- **Issue:** Hardcodes `"en-IN"` and `"INR"` instead of importing from `@/shared/constants/locale` (`MONEY_LOCALE`, `CURRENCY_CODE`).
- **Fix:** Import the shared constants.

### 7. Unawaited `registration.update()` in PWA registration

- **File:** `src/platform/pwa/register.ts:43-49`
- **Issue:** `registration.update()` returns a Promise that isn't awaited; the `waiting` check immediately after may return stale status.
- **Fix:** `await registration.update()` before checking `registration.waiting`.

### 8. Boundary violations — budgets/recurring import from transactions domain internals

- **Files:**
  - `src/features/budgets/components/budget-detail.tsx:13-16`
  - `src/features/budgets/components/budget-list.tsx:18-21`
  - `src/features/recurring/components/recurring-rule-list.tsx:13-16`
- **Issue:** Import `formatCurrency`/`formatDate` directly from `@/features/transactions/domain/format` instead of going through shared utilities.
- **Fix:** Move formatters to `@/shared/domain/format` or import through the transactions barrel (`@/features/transactions`).

### 9. No explicit authorization on mutation server actions

- **Issue:** All server actions delegate ownership checks to RLS. No application-level guard ensures the authenticated user owns the resource they're mutating. If RLS is ever misconfigured or bypassed (e.g., service role key used), the guard is absent.
- **Fix:** Add explicit `userId` checks before every mutation.

## 🔵 Low Severity

### 10. `Money.format()` defaults to integer rounding

- **File:** `src/shared/domain/value-objects/money.ts:56-62`
- **Issue:** `maximumFractionDigits = 0` rounds `₹10.50` to `₹11`.
- **Fix:** Change default to `2` or make it locale-aware.

### 11. `EntityId.create()` is public

- **File:** `src/shared/domain/value-objects/entity-id.ts:9-17`
- **Issue:** Public static method bypasses subclass factories like `AccountId.from()`.
- **Fix:** Change visibility to `protected`.

### 12. Subclass `from()` creates and discards `EntityId`

- **Files:**
  - `src/features/accounts/domain/value-objects/account-id.ts:9`
  - `src/features/transactions/domain/value-objects/transaction-id.ts:9`
  - `src/features/categories/domain/category-id.ts:9`
  - `src/shared/domain/value-objects/entity-id.ts:37`
- **Issue:** `AccountId.from(value)` calls `EntityId.create(value, "account").value` — creates an unused `EntityId` just to extract `.value`, then passes it to `new AccountId(value)`.
- **Fix:** Inline the validation logic or make the subclass constructor accept raw strings.

### 13. `fieldErrorsFromZod` drops nested and subsequent errors

- **File:** `src/shared/application/zod-errors.ts:3-12`
- **Issue:** Only captures `issue.path[0]`, first error wins per field. SuperRefine issues with nested paths and subsequent field errors are silently lost.
- **Fix:** Join multiple errors per field with `; ` or append to an array.

### 14. `restoreTransaction` doesn't set `updated_at`

- **File:** `src/features/transactions/infrastructure/supabase-transaction.repository.ts:99-122`
- **Issue:** Restored transactions have stale `updated_at` timestamps.
- **Fix:** Set `updated_at` to current timestamp on restore.

### 15. Negative budget amounts shown as "safe"

- **File:** `src/features/budgets/domain/usage.ts:143`
- **Issue:** `budget.amount > 0 ? (spent / budget.amount) * 100 : 0` masks negative amounts as zero percentage (status = "safe"). While the schema validates `amount > 0`, the function doesn't assume this invariant.
- **Fix:** Let negative amounts produce negative percentage, or add an assertion at the function boundary.

## 🏛️ Architectural Design Issues

### 16. 12 pass-through use case classes with zero business logic

- **Files:** `features/{accounts,categories,budgets,recurring}/application/use-cases/*.use-case.ts`
- **Issue:** Account, category, budget, and recurring simple CRUD use cases are one-liner delegations to repositories. They add no validation, transformation, or orchestration — just pass input straight through to `this.repository.create({...input, userId})`. 12 structurally identical classes with no business value.
- **Fix:** Remove thin use cases for CRUD-only domains; server actions can call repositories directly. Keep only use cases with actual business logic (e.g., `ExecuteRecurringRuleUseCase`, `CreateTransactionUseCase`).

### 17. Components tightly coupled to `useActionState`

- **Files:** 6 form components across `features/{accounts,categories,transactions,recurring,budgets,auth}/components/`
- **Issue:** All form components use the `useServerActionForm` hook (a 38-line wrapper around `useActionState`). Components cannot be rendered without mocking React's `useActionState` — the coupling is baked into the component signature. Accepting `state` + `isPending` as props would decouple presentation from form state management.
- **Fix:** Change component interfaces to accept `state` and `isPending` as props. Create a thin container component (or the page itself) that wires the server action through `useActionState` and passes the result down.

### 18. `Account.restore()` requires 12 fields to construct

- **File:** `src/features/accounts/domain/entities/account.ts`
- **Issue:** The entity's private constructor forces all callers (including tests) to use `Account.restore()`, which requires a full persistence snapshot with 12 fields (id, name, type, balance, color, icon, is_active, created_at, updated_at, deleted_at, user_id, include_in_total). Even when testing simple domain rules like balance constraints, every test must construct a valid full snapshot.
- **Fix:** Add a `Account.create(name, type, balance)` static factory that only requires business-relevant fields. Keep `restore()` for the persistence adapter. This decouples domain tests from persistence shape.
